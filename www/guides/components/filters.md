# Фильтры (каталога / внутри карточек товаров) {#filters}

<ul class="info-block">
  <li class="info-block-itm">Тип компонента: Навигация листинга</li>
  <li class="info-block-itm">SEO-риск: Высокий </li>
  <li class="info-block-itm">Критично для: Индексации, краулингового бюджета, дублей страниц</li>
</ul>

## Суть проблемы {#point}

Фильтры в создают новые состояния страницы, которые могут иметь собственные URL. Отсутствие <a href="/guides/components/filters-base.html#rules">стратегии</a> и <a href="/guides/components/filters-base.html#url-control">контроля над URL</a> будет приводит к:

- неконтролируемой генерации ненужных URL
- дублям страниц
- перерасходу краулингового бюджета (или лимиту обхода)
- попаданию в индекс нецелевых страниц

## Правильная модель {#correct-model}

- Фильтры - это "генератор страниц", а не просто UI.
- Каждая комбинация фильтров - потенциально новая SEO-страница, которую нужно либо индексировать (<strong>белый список</strong>), либо закрывать от индексации.
- Фильтры должны формировать предсказуемые URL.
- Индексироваться должны только заранее определённые комбинации фильтров.

## Пример {#example}

::: code-group

```HTML [Каталог]
<!-- Фильтр для категории "laptops" -->

<form method="GET" action="/catalog/laptops/" class="filters-form">
  <fieldset>
    <legend>Бренд</legend>
    <label>
      <input type="checkbox" name="brand" value="apple"> Apple
      </label>
    <label>
      <input type="checkbox" name="brand" value="asus"> ASUS
      </label>
    <label>
      <input type="checkbox" name="brand" value="dell"> Dell
      </label>
    <label>
      <input type="checkbox" name="brand" value="lenovo"> Lenovo (UX only)
      </label>
  </fieldset>

  <fieldset>
    <legend>Оперативная память</legend>
    <label>
      <input type="checkbox" name="ram" value="8"> 8GB (UX only)
    </label>
    <label>
      <input type="checkbox" name="ram" value="16"> 16GB
    </label>
  </fieldset>

  <button type="submit">Применить</button>
</form>

```

```HTML [Карточка]
<!-- Фильтр в карточке товаров -->
<form method="GET" action="/product/iphone-10/">

  <select name="color">
    <option value="black">Чёрный</option>
    <option value="white">Белый</option>
  </select>

  <button type="submit">Выбрать</button>

</form>
```

:::

Форма отправляется GET-запрос с параметрами в URL, а сервер возвращает HTML страницы с соответствующим содержанием.

```HTML
/product/iphone-10/?color=black
```

<p>В результате такого запроса:</p>
<ul>
  <li>возвращается полностью отрендеренная HTML-страница</li>
  <li>с выбранным вариантом (черный)</li>
  <li>без необходимости использовать JS</li>
</ul>

<div class="text-block text-block--tip mt-30">
  <p>Фильтр должен:</p>
  <ul>
    <li>иметь action → чтобы формировался URL и все работало без JS</li>
    <li>работать без JS → базовое требование для SEO и доступности</li>
  </ul>
</div>

## Некорректные запросы {#incorrect-requests}

<p>Через фильтр (или вручную) могут формироваться комбинации параметров, для которых не будет соответствующего контента.</p>

```HTML
<!-- Например, диапазон цены слишком низкий -->
/catalog/laptops/?brand=lenovo&price=100-300

<!-- Или не существующие параметры в URL -->
/catalog/laptops/?brand=lenovo&color=pinkfloyd
```

<p>Обрабатывать такие запросы можно тремя способами:</p>

<ol>
  <li class="mt-30">
  <p>Редирект на базовую страницу категории / подкатегории.</p>
  <p>Безопасно для SEO, нет ошибок 404, URL без контента не индексируется. Рекомендуемый вариант для большинства случаев.</p>

```HTML
<!-- Бэк возвращает страницу категории -->
/catalog/laptops/

```

  </li>
  <li class="mt-30">
    <p>Редирект на 404.</p>
    <p>Плохо для SEO, особенно если такие комбинации генерируются формой. Это приведет к большому количеству "мусорных" 404.</p>

```HTML
<!-- Бек возвращает страницу 404 -->
/error-page/

```

  </li>
  <li class="mt-30">
    <p>Редирект на страницу с пустым результатом, но с <i>200 OK</i>.</p>
    <p>Возвращается корректная страница с сообщением "товары не найдены". Обязательно применять <code>noindex</code>, чтобы не индексировать пустые страницы. Допустимо для SEO.</p>

```HTML
<!-- Бек возвращает страницу "товары не найдены" -->
/sorry-page/
```

  </li>

</ol>

<p class="mt-40"><strong>Рекомендации для работы с некорректными запросами:</strong></p>

- <strong>Переход на базовую категорию</strong> - чаще всего самый безопасный вариант.
- <strong>Контроль параметров</strong> - фильтры на фронте должны предлагать только допустимые комбинации.
- <strong>Noindex для несуществующих комбинаций</strong> - пользователь или бот всегда могут попасть на URL с пустым результатом.
- <strong>Canonical на главную категорию</strong> - если комбинация пустая или неключевая.

## Стратегия {#rules}

<div class="text-block text-block--info mt-30">
  <p>Фильтры могут создавать неограниченное количество URL, но количество индексируемых страниц должно быть строго ограничено и контролироваться.</p>
</div>

<ol>
  <li>Белый список (SEO)
    <ul class='ml-30'>
      <li>URL индексируются</li>
      <li>есть в sitemap</li>
      <li>есть в перелинковке</li>
      <li>имеют уникальный контент (или хотя бы релевантный)</li>
      <li>имеют уникальный <code>canonical</code></li>
    </ul>
  </li>

 <li>Серый список (UX)
  <ul class='ml-30'>
    <li>генерируются через form + SSR</li>
    <li>доступны по URL</li>
    <li>отсутствуют в перелинковке</li>
    <li>содержат <code>canonical</code> на ближайшую релевантную категорию или SEO-страницу (<a href="/docs/technical/canonical.html#resolution">как выбрать canonical</a>)</li>
  </ul>

  </li>

  <li>Закрываем от индексации мусорные страницы через `robots.txt` или через <a href="/guides/technical/x-robots-tag.html">meta="robots"</a>.

  </li>
</ol>

## Контроль URL {#url-control}

<div class="text-block text-block--info mt-30">
  <p>Важно понимать, что фильтр и перелинковка тесно связаны, и всё вертится вокруг контроля над URL. Поэтому сначала определяется стратегия (списки: white, gray, black), а затем следует реализация фильтров и перелинковки.</p>
</div>

<ol>
  <li><strong>Списки - слой управления индексацией (SEO-контроль)</strong>
    <ul>
      <li><strong>Белый список</strong> → индексируем + линкуем</li>
      <li><strong>Серый список</strong> → не индексируем (canonical / noindex)</li>
      <li><strong>Чёрный список</strong> → блокируем</li>
    </ul>
  </li>
  <li>
    <p><strong>Фильтр - это UX-слой</strong></p>
    <p>Фильтр должен генерировать любые комбинации (в рамках логики каталога) и позволять пользователям свободно с ними взаимодействовать. Фильтр НЕ ограничивается списками. Иначе говоря - <i>фильтр шире, чем SEO-стратегия</i>.</p>
  </li>
  <li><strong>Перелинковка - строится ТОЛЬКО на белом списке</strong>
    <ul>
      <li>страницы из белого списка участвуют в перелинковке</li>
      <li>страницы из серого и черного списка не получают ссылки</li>
    </ul>

  </li>
</ol>

<div class="text-block text-block--tip mt-30">
  <p>Основное правило:</p>
    <ul>
      <li>Фильтр отвечает за UX - генератор всех состояний.</li>
      <li>Списки отвечают за SEO - SEO-фильтр над состояниями.</li>
      <li>Перелинковка отвечает за вес - только для выбранных URL.</li>
    </ul>
</div>

## Списки URL {#lists}

<ol>
  <li><strong>Белый список</strong> - индексируемые URL
    <ul>
      <li>имеют поисковый спрос (семантика)</li>
      <li>отвечают на уникальный интент</li>
      <li>могут ранжироваться как отдельные страницы</li>
    </ul>

```HTML
<!-- Категории (обязательно) -->
/catalog/laptops/

<!-- Бренды (если есть спрос) -->
/catalog/laptops/apple/
/catalog/laptops/?brand=apple

<!-- 1 интент (ограниченно и только если есть спрос) -->
/catalog/laptops/?ram=16gb

<!-- Жёстко заданные SEO-комбинации -->
<!-- выбираются вручную, и ТОЛЬКО если есть спрос -->
/catalog/laptops/apple-16gb/
/catalog/laptops/gaming/
```

  </li>
  <li><strong>Серый список</strong> - URL не индексируем, но оставляем доступными
    <ul>
      <li>узкие выборки (нет спроса)</li>
      <li>не нужны в поиске (технические фильтры, диапазоны и т.д.) </li>
      <li>но полезны для пользователей</li>
    </ul>

```HTML
<!-- Много комбинаций фильтров -->
/catalog/laptops/?brand=apple&ram=16gb&price=1000-2000

<!-- Технические фильтры -->
?sort=price_asc
?in_stock=true
?page=2

<!-- Диапазоны -->
?price=1000-2000
```

  </li>
  <li><strong>Черный список</strong> - полностью блокируемые URL
    <ul>
      <li>бесполезные для пользователя</li>
      <li>мусорные комбинации</li>
      <li>создающие ловушки для краулеров</li>
    </ul>
  </li>
</ol>
````

## Как не нужно {#how-not-to}

<ul class="list-reset">
  <li>❌ Фильтры работают только через JS (без URL)</li>
  <li>❌ URL не меняется при применении фильтра</li>
  <li>❌ Контент подгружается только через API</li>
  <li>❌ Генерируются бесконечные комбинации URL</li>
  <li>❌ Нет canonical / noindex стратегии</li>
</ul>

## Чек-лист {#checklist}

<ul class="list-reset">
  <li>✔️ Есть стратегия: какие страницы индексируются</li>
  <li>✔️ Количество индексируемых страниц ограничено</li>
  <li>✔️ SEO-важные результаты фильтра имеют свою страницу</li>
  <li>✔️ Фильтры работают без JavaScript</li>
  <li>✔️ При применении фильтра создаётся новый URL</li>
  <li>✔️ Результат фильтра доступен при прямом переходе по URL</li>
</ul>

### Инструменты для проверки {#tools-for-checking}

- DevTools → View Page Source
- DevTools → Network
- отключение JavaScript в браузере
