# Чек-лист: Core Web Vitals (основные веб-показатели)

<div class="text-block">

- Сайт считается соответствующим требованиям Core Web Vitals, если больше 75% пользовательских загрузок страниц укладываются в пороги LCP, CLS и INP
- Для технического SEO важно контролировать реальные пользовательские данные

</div>

<!-- --------- -->

## 1. Проверка LCP (Largest Contentful Paint) {#1}

<div class="text-block">
  <p><code>Largest Contentful Paint (LCP)</code> - скорость загрузки основного контента страницы.</p>
</div>

<ul class='list-reset'>
  <li>1-1. LCP ≤ 2.5s на мобильных устройствах</li>
  <li>1-2. Основной LCP-элемент (hero image, баннер, заголовок) определяется корректно</li>
  <li>1-3. LCP-ресурс не загружается через <code>lazy-load</code></li>
  <li>1-4. LCP-изображение имеет <code>preload</code> или высокий приоритет загрузки</li>
  <li>1-5. Влияющий на LCP-элемент <code>CSS</code> - не блокирует рендеринг</li>
  <li>1-6. Нет тяжелых JS-бандлов, задерживающих отрисовку</li>
</ul>

<div class="text-block text-block--tip mt-30">
  <p>Главный визуальный элемент (первый экран) страницы должен загружаться и отображаться максимально рано - без блокирующих <code>CSS/JS</code> и без <code>lazy-loading</code>.</p>
</div>

<!-- --------- -->

## 2. Проверка CLS (Cumulative Layout Shift) {#2}

<div class="text-block">
  <p><code>Cumulative Layout Shift (CLS)</code> - насколько элементы страницы «прыгают» при загрузке (визуальная стабильность верстки).</p>
</div>

<ul class='list-reset'>
  <li>2-1. У изображений указаны width и height или aspect-ratio</li>
  <li>2-2. Рекламные блоки и iframe имеют зарезервированное место</li>
  <li>2-3. Динамический контент не сдвигает существующие элементы</li>
  <li>2-4. Web fonts не вызывают скачки layout (используется font-display)
  
  ```CSS
  @font-face {
  font-family: "Inter";
  src: url("/fonts/inter.woff2") format("woff2");
  font-display: swap; /* вот это */
  }
  
  ```
</li>
</ul>

<div class="text-block text-block--info mt-30">
  <p>Основные значения font-display:</p>
  <p><code>swap</code> - текст показывается сразу fallback-шрифтом, потом меняется. Хорошо для производительности и SEO.</p>
  <p><code>block</code> - текст временно скрывается, пока не загрузится шрифт. Плохо для UX.</p>
  <p><code>fallback</code> - короткая попытка загрузить шрифт, потом fallback.</p>
  <p><code>optional</code> - браузер может вообще не загружать шрифт, если сеть медленная.</p>
</div>

<div class="text-block text-block--tip mt-30">
  <p>Страница должна быть визуально стабильной - элементы не смещаются во время загрузки.</p>
</div>

<!-- --------- -->

## 3. Проверка INP (Interaction to Next Paint) {#3}

<div class="text-block">
  <p><code>Interaction to Next Paint (INP)</code> - отзывчивость интерфейса (как быстро страница реагирует на действия пользователя).</p>
</div>

<ul class='list-reset'>
  <li>3-1. INP ≤ 200 ms</li>
  <li>3-2. Клик по кнопкам и ссылкам обрабатывается без заметной задержки</li>
  <li>3-3. Нет тяжелых JS-задач → тяжёлые вычисления, массовые операции с DOM, большой JSON-парсинг, большая гидрация в SPA (часто в Vue, React, Nuxt, Next). Рекомендуемое значение - менее 50 ms.</li>
  <li>3-4. Минимизировано количество JS на странице</li>
  <li>3-5. Основные пользовательские действия (меню, фильтры, формы) работают быстро</li>
</ul>

<div class="text-block text-block--tip mt-30">
  <p>После взаимодействия пользователя интерфейс должен реагировать почти мгновенно.</p>
</div>

<!-- --------- -->

## 4. Проверка мобильного рендеринга {#4}

<ul class='list-reset'>
  <li>4-1. Core Web Vitals приемлемы у mobile версии</li>
  <li>4-2. Производительность стабильна на реальных устройствах</li>
  <li>4-3. Нет тяжелых мобильных скриптов (аналитика, виджеты, чаты)</li>
</ul>

<div class="text-block text-block--tip mt-30">
  <p>Метрики должны проходить пороговые значения на мобильных устройствах, так как поисковые системы используют mobile-first индексацию.</p>
</div>

<!-- --------- -->

## 5. Проверка по типам страниц {#5}

<ul class='list-reset'>
  <li>5-1. Главная страница</li>
  <li>5-2. Категории</li>
  <li>5-3. Карточки товаров / услуг</li>
  <li>5-4. Контентные страницы (статьи, блог)</li>
</ul>

<div class="text-block text-block--tip mt-30">
  <p>Core Web Vitals должны соответствовать требованиям на всех ключевых шаблонах страниц.</p>
</div>

## Подводные камни {#hints}

- <strong>Лабораторные тесты vs реальные данные.</strong> Лабораторные тесты (Lighthouse) могут показывать хорошие показатели, но реальные пользователи могут иметь плохие CWV.
- <strong>Разные страницы - разные метрики.</strong> Главная может проходить CWV, но категории или карточки товаров - нет.
- <strong>Кэш и CDN.</strong> При повторных тестах метрики могут быть лучше из-за кэша.
- <strong>География тестирования.</strong> Метрики могут сильно отличаться при тестировании из разных регионов.
- <strong>A/B тесты и виджеты.</strong> Маркетинговые скрипты, чаты, popups часто ухудшают INP и LCP.
- <strong>Динамический контент.</strong> Ленивая подгрузка элементов может вызывать CLS при неправильной реализации.
- <strong>Фильтры и SPA-интерфейсы.</strong> На e-commerce страницах INP часто ухудшается из-за тяжелых JS-операций.
