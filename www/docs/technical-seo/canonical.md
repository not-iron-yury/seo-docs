# Canonical {#canonical}

<!-- --- -->

<div class="text-block">
  <p><code>rel="canonical</code> - это указание поисковой системе, какая версия страницы является основной (предпочтительной) при наличии дубликатов.</p>
  <p>Сигнал сообщает: «Эти страницы могут быть похожи или идентичны. Индексируй и учитывай только эту версию».</p>
  <p>Без canonical поисковая система самостоятельно выбирает каноническую версию, что не всегда совпадает с намерением владельца сайта.</p>
</div>

Пример:

```html
<link rel="canonical" href="https://example.com/catalog" />
```

<!-- --- -->

## Как используют поисковые системы {#how}

### 🔹 Google {#how-google}

- Canonical - это рекомендация, а не директива.
- Если контент существенно различается - Google может игнорировать тег.
- Если есть конфликтующие сигналы (редирект, sitemap, внутренняя перелинковка), Google выбирает свою версию.

### 🔸 Яндекс {#how-yandex}

- В целом учитывает `rel="canonical"`
- Может игнорировать при сильных расхождениях контента
- В коммерческих нишах чувствителен к дублированию параметрических страниц

<!-- --- -->

## Практические рекомендации {#recs}

### 1. Общие {#recs-1}

- На всех индексируемых страницах должен быть `rel="canonical"`.
- Canonical должен вести на страницу со статусом 200.
- Canonical-URL индексируется (не noindex, не закрыт в robots.txt).
- В `rel="canonical"` используются абсолютные URL.

### 2. Для e-commerce / фильтров {#recs-2}

- Страницы сортировки → canonical на основную категорию.
- Параметры (?sort=, ?filter=) → canonical или noindex.
- Пагинация → canonical на саму страницу пагинации (не всегда на первую).

### 3. Согласованность сигналов {#recs-3}

Canonical должен совпадать с:

- версией в sitemap
- внутренними ссылками
- HTTPS-версией
- основным доменом (www / без www)

## Типичные ошибки {#mistakes}

<div class="text-block  text-block--warning">
  <p>Неправильное использование не приводит к прямым санкциям поисковых систем.  
  Однако могут быть серьёзные последствия - избыточная индексация мусорных URL или исключение важных страниц из индекса.</p>
</div>

### 1. Размывание ссылочного веса {#mistake-1}

Если 5 дублей без canonical:

- ссылочный вес распределяется между ними
- ни одна страница не усиливается достаточно

### 2. Неправильная каноникализация {#mistake-2}

- Canonical указывает на нерелевантную страницу → потеря позиций
- Canonical указывает на главную со всех страниц → деиндексация каталога
- Циклические canonical → робот не может определить приоритет

### 3. Игнорирование canonical {#mistake-3}

Поисковик выберет версию сам, если:

- контент слишком разный
- указана другая структура в sitemap
- есть противоречия

<div class="text-block text-block--info mt-40">
Когда дубль - ошибка?

- Один и тот же контент доступен по разным URL без canonical и без редиректов
- CMS генерирует лишние URL
- Страницы отличаются только мелкими деталями (регистром, слешами, якорями)
</div>

## Влияние на другие факторы {#factors}

### 1. Ссылочные факторы {#factors-1}

Canonical объединяет:

- внешние ссылки
- внутренние ссылки
- анкорный вес (влияние текста ссылки на ранжирование)

Поэтому неправильная настройка может привести к потери веса.

### 2. Контентные факторы {#factors-2}

Дубли:

- снижают уникальность
- создают конкуренцию страниц внутри сайта
- усложняют ранжирование

### 3. Поведенческие факторы {#factors-3}

Если пользователи попадают на разные версии URL:

- данные о CTR и взаимодействии рассеиваются
- ухудшается консолидация поведенческих сигналов

## Корректный дубль страницы {#correct}

<div class="text-block text-block--tip">
  <p>
  Если у страницы есть самостоятельная ценность в поиске - она должна быть индексируемой. И наоборот.
  </p>
</div>

### 1. URL с параметрами {#correct-1}

```text
/catalog
/catalog?sort=price
/catalog?utm_source=google
```

Контент по сути один и тот же. Это технический дубль, который:

- либо канонизируется на `/catalog`
- либо закрывается от индексации
- либо нормализуется через редиректы

### 2. Версия для печати {#correct-2}

```text
/article
/article?print=1
```

Допустимо, но:

- print-версия → canonical на основную
- в sitemap не добавляется

### 3. HTTP / HTTPS {#correct-3}

Одна версия должна редиректиться `301`.
Canonical - это запасной механизм, а не основной.

### 4. Фильтры и фасетная навигация (e-commerce) {#correct-4}

```text
/shoes?color=red
/shoes?size=42
```

Тут тонкий момент:

- если страница имеет поисковый спрос → тогда может быть самостоятельной SEO-страницей
- если нет → canonical на базовую категорию

## Матрица решений {#resolution}

| Тип страницы                    | Canonical   | Sitemap | Индексация |
| ------------------------------- | ----------- | ------- | ---------- |
| SEO-страницы                    | на себя     | Да      | Да         |
| Параметры с SEO-ценностью       | на себя     | Да      | Да         |
| Параметры без SEO-ценности      | на канон    | Нет     | Нет        |
| Технический дубль               | на канон    | Нет     | Нет        |
| UX-страницы (корзина, checkout) | опционально | Нет     | Нет        |
| Админка / закрытые страницы     | не нужен    | Нет     | Нет        |

Совет для SEO-ценных страниц:

```HTML
<!-- Параметры с SEO-ценностью -->
/catalog/hats?color=red
/catalog/hats?color=white
<!-- хуже ранжируется (в среднем), ниже CTR, сложнее продвигать как посадочную -->

<!-- Лучше заменить на ЧПУ -->
/catalog/hats/red/
/catalog/hats/white/
<!-- проще контролировать индексацию, чище для SEO, лучше CTR и ранжирование -->
```

## Практический вывод {#сonclusion}

1. Делать нормализацию URL (слеши, регистр, www)
2. Контролировать генерацию параметров
3. Не давать CMS плодить индексируемые мусорные URL
4. Sitemap генерировать строго из каноничных сущностей

  <p>Redirect - это архитектурное решение.</p>
</div>
